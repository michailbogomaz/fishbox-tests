# Инструкции fast — Тип C (полные файловые проверки)

1. **Подтверждение инструкций.** После завершения типов A и B выведи блок  
   `Подтверждение инструкций — [./.cursor/checklist-instrusctions_fast_type-c.md#L1](./.cursor/checklist-instrusctions_fast_type-c.md#L1)`  
   и явно укажи, что файл прочитан полностью.
2. **Язык.** Все ответы пользователю — только на русском.
3. **Подготовка:**
   - используй ранее собранный список файлов из `git log` и не меняй порядок;
   - уточни интерфейс (REST/GraphQL); GraphQL-пункты можно пропускать, если проект чисто REST;
   - загрузить чеклист `./.cursor/checklist-type-c.md`.
4. **Общие правила:**
   - обработка строго по порядку файлов из списка;
   - для каждого файла сначала определить, какие пункты применимы (по расширению, назначению, контексту);
   - применять пункт → читать файл целиком (`read_file` без `limit`);
   - если пункт неприменим — ставь ⏭️ и коротко объясняй причину;
   - не переходить к следующему файлу, пока текущий полностью не закрыт.
5. **Структура отчёта на файл:**
   ```
   Файл: path/to/file.ts
   - Пункт N (тип C) — [./.cursor/checklist-type-c.md#L<номер>](./.cursor/checklist-type-c.md#L<номер>)
     Запускаю команду `read_file path/to/file.ts`
     Результат: …
     Статус: ✅/❌/⏭️ (причина)
   ```
   Если для пункта нужна отдельная команда (например, `rg` по проекту), укажи её внутри блока.
6. **Команды:** перед `read_file`/`codebase_search`/`rg`/`python` всегда выводи строку `Запускаю команду \`...\``. После выполнения — краткий результат.
7. **Оптимизация:**
   - если файл не затрагивает большинство пунктов (например, миграция или env), сразу помечай неприменимые пункты;
   - выявив проблему и поправив её (если `autofix=1`), повторяй только затронутый пункт.
8. **Финал по типу C:** после обработки всех файлов дай сводку статусов по каждому пункту и перечисли найденные нарушения/рекомендации.
